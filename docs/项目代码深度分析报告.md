## 一、项目整体架构与技术栈分析

### 1.1 架构图逆向推导

基于代码逆向分析，项目采用 **9层分层架构**：

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 接入层    │ Lark Bot Webhook、Callback、File Upload     │
│              │ lark_webhook.py (514行) + lark_callback.py │
├─────────────────────────────────────────────────────────────┤
│ 2. API网关层  │ FastAPI、验签、幂等、trace_id、异常处理      │
│              │ router.py + middleware.py + exceptions.py     │
├─────────────────────────────────────────────────────────────┤
│ 3. 会话上下文层│ ContextService、对话状态机、多轮补参         │
│              │ context_service.py + dialog_state.py          │
├─────────────────────────────────────────────────────────────┤
│ 4. 权限隔离层  │ 用户-项目权限、群-项目绑定、Skill权限校验     │
│              │ user_project_role.py + group_project_binding  │
├─────────────────────────────────────────────────────────────┤
│ 5. 编排层     │ Intent Router、Skill Registry、Orchestrator  │
│              │ orchestrator.py (906行) + intent_router.py     │
├─────────────────────────────────────────────────────────────┤
│ 6. AI能力层   │ LLM Gateway、Prompt Manager、RAG、安全防护    │
│              │ llm_gateway.py (723行) + retriever.py (554行) │
├─────────────────────────────────────────────────────────────┤
│ 7. Skill插件层 │ 13个核心Skill，BaseSkill规范，manifest       │
│              │ base.py + 13个Skill实现                        │
├─────────────────────────────────────────────────────────────┤
│ 8. 集成适配层  │ 飞书、项目管理、财务、DevOps、OA适配器    │
│              │ lark/client.py + lark/service.py            │
├─────────────────────────────────────────────────────────────┤
│ 9. 数据层     │ PostgreSQL、pgvector、Redis、Celery          │
│              │ repositories/base.py (488行) + 14个ORM模型     │
└─────────────────────────────────────────────────────────────┘
```

**代码引用**: 核心编排引擎 `app/orchestrator/orchestrator.py` 第1-906行实现了完整的消息处理管线。

### 1.2 技术栈选型评估

| 技术 | 选型 | 评价 |
|------|------|------|
| 语言 | Python 3.11 | ✅ 合理，AI生态最成熟 |
| Web框架 | FastAPI 0.115.x | ✅ 异步原生、类型安全 |
| ORM | SQLAlchemy 2.0.x (async) | ✅ 现代异步ORM最佳选择 |
| 数据库 | PostgreSQL 16 + pgvector | ✅ RAG场景需要向量检索 |
| 缓存 | Redis 7.4-alpine | ✅ 行业标准 |
| 消息队列 | RabbitMQ 4.0-management | ✅ 管理面板便于运维 |
| 异步任务 | Celery 5.4.x | ✅ 成熟稳定 |
| HTTP客户端 | httpx | ✅ 异步HTTP库首选 |
| 配置管理 | pydantic-settings | ✅ 类型安全的配置管理 |

**技术栈评分: A-**

**优势**:
- 全部选用异步原生库，无阻塞风险
- pgvector 集成省去独立向量数据库（如 Milvus/Chroma）的运维成本
- httpx 替代 requests，全面异步化

**风险点**:
- pgvector 在 PostgreSQL 上性能不如专用向量数据库（Milvus 在百万级向量场景下快10x）
- Celery 单进程模型下并发能力受限于 Worker 数量

### 1.3 分层架构设计评估

**亮点**:
1. **分层清晰**: 9层架构职责明确，每层依赖下一层，无循环依赖
2. **仓储模式**: `ProjectScopedRepository` 强制 project_id 过滤，防止跨项目数据泄露
3. **插件化Skill**: BaseSkill 抽象基类 + SkillManifest 注册机制，符合开闭原则

**问题诊断**:

**P1 - 全局单例模式过度使用**: 多个模块使用 `global _instance` + `get_instance()` 模式：
- `app/integrations/lark/client.py:428` → `global _lark_client`
- `app/ai/llm_gateway.py:715` → `global _llm_gateway`
- `app/rag/retriever.py:539` → `global _vector_retriever`
- `app/orchestrator/skill_registry.py:141` → `global _skill_registry`
- `app/orchestrator/intent_router.py:507` → `global _intent_router`

**影响**: 测试困难（状态不可重置）、多租户场景可能出现问题、热重载不生效。

**P2 - 依赖注入缺失**: 所有服务都通过全局函数获取实例，非依赖注入。FastAPI 原生支持 `Depends()`，但未使用。

## 二、前端代码深度分析

### 2.1 交互界面分析（飞书卡片）

项目**无传统前端**，用户交互完全通过**飞书消息卡片**实现。

**卡片类型**（基于 `app/integrations/lark/schemas.py` 中的 `LarkCardBuilder`）:
1. **文本卡片**: 简单通知、确认消息
2. **Markdown卡片**: 项目总览、周报等富文本展示
3. **交互卡片 (interactive_card)**: 交互式卡片，包含按钮、表单、下拉等交互组件

**关键代码引用**:
```python
# app/integrations/lark/client.py:262-288

    chat_id: Optional[str] = None,

    """发送交互卡片消息."""
    data = {
        "msg_type": "interactive",
        "receiver_id": user_id,

    }
```

### 2.2 卡片组件设计评估

**当前实现模式**: 各 Skill 自行构建卡片内容，无统一的卡片组件库。

**问题诊断**:

**P1 - 卡片构建分散**: 每个 Skill 直接构造字典作为卡片配置，无封装。
- 例如 `weekly_report_skill.py` 中直接构建 `template_card` 字典
- 缺少 `CardComponent` 抽象层，导致卡片样式不一致

**P2 - 无卡片版本管理**: 飞书卡片模板支持版本管理，但代码中未实现。

**P2 - 无卡片渲染器测试**: `app/presentation/cards/base.py` 和 `app/presentation/renderers.py` 存在但缺乏卡片视觉效果的测试。

### 2.3 用户交互流程分析

基于 `app/orchestrator/orchestrator.py` 的完整交互流程：

```
用户发送消息 → Lark Webhook (5秒内响应"success")
  ↓
Background Task: _process_message_async()
  ↓
消息幂等检查 → Redis去重
  ↓
ContextService.build_user_context() → 构建用户上下文
  ↓
DialogStateMachine.get_or_create_session() → 获取/创建会话
  ↓
根据状态分发:
  - IDLE → IntentRouterV2.recognize_with_context()
  - PARAM_COLLECTING → 补参逻辑
  - CONFIRMATION_PENDING → 确认执行
  - EXECUTING → 拦截新消息
  ↓
Skill执行 → 构建飞书卡片 → 发送给用户
```

**问题诊断**:

**P2 - 超时风险**: 飞书要求5秒内响应。当前使用 `BackgroundTasks` 异步处理，但如果 LLM 调用耗时过长（如 30s），用户在此期间无反馈，体验差。

**建议**: 增加"处理中"中间状态卡片（如"正在为您生成周报，预计需要15秒..."）。

```python
# app/repositories/base.py:235-261
async def get_by_id_with_project(
    self,
    id: uuid.UUID,
    project_id: uuid.UUID,
) -> Optional[ModelType]:
    """根据ID和project_id查询记录. 强制检查project_id."""
    result = await self.session.execute(
        select(self.model).where(
            and_(
                self.model.id == id,
                self.model.project_id == project_id,
            )
        )
    )
    return result.scalar_one_or_none()
```

**设计评价: A-**

**优势**:
- ✅ 全部使用 UUID 主键（`uuid.uuid4`）
- ✅ 审计字段统一通过 `AuditMixin`（`created_at`, `updated_at`, `created_by`）
- ✅ 合理的索引设计（project_id、status 等高频查询字段）

**问题诊断**:

**P2 - 缺少软删除**: 部分模型缺少 `is_deleted` 或 `deleted_at` 字段，物理删除可能导致数据丢失。

**P2 - 审计字段未统一**: 虽然 `AuditMixin` 存在，但部分模型手动定义时间字段，未完全复用 Mixin。

### 4.2 SQL性能分析

**正面实践**:
- `app/repositories/base.py:329` → `select(self.model).where(self.model.project_id == project_id).offset(skip).limit(limit)` ✅ 分页查询
- `app/repositories/base.py:378` → `select(func.count()).select_from(self.model).where(...)` ✅ 计数查询使用聚合函数

**问题诊断**:

**P2 - RAG向量检索性能**: `app/rag/retriever.py:176-185` 中，稠密检索先加载**所有**有 embedding 的文档到内存，再逐条计算余弦相似度：

```python
# app/rag/retriever.py:176-185
result = await self.session.execute(query)
docs = result.scalars().all()
for doc in docs:
    similarity = self._cosine_similarity(query_embedding, doc.embedding)
```

**性能风险**: 当 `knowledge_documents` 表超过 1000 条时，全表扫描 + Python 计算相似度将严重拖慢响应。应使用 PostgreSQL pgvector 的 `<=>` 算子在数据库侧计算相似度并 LIMIT。

**P2 - 稀疏检索使用 LIKE**: `app/rag/retriever.py:251-255` 使用 `LIKE '%term%'` 全表扫描，无法利用索引。

### 4.3 事务管理评估

**当前实现**: 直接使用 `AsyncSession`，无显式事务管理。

**问题诊断**:

**P1 - 无显式事务边界**: 代码中未见 `async with session.begin():` 或 `async with session.begin_nested():`（Savepoint）。在多表操作场景下（如创建项目同时创建角色绑定），如果中途失败可能导致数据不一致。

**P1 - 无数据库级约束**: 虽然有 ORM 层的 project_id 过滤，但数据库层面缺少外键约束（如 `task.project_id → project.id` 的 FK），可能导致孤儿记录。

### 4.4 数据备份方案

**未发现**: 代码中无备份策略、无数据导出工具、无迁移脚本。

**建议**:
- 生产环境应配置 PostgreSQL 定时 pg_dump
- 向量数据备份需特别注意 pgvector 扩展兼容性
- 建议实现 `manage.py backup` 命令

### 4.5 敏感数据处理评估

**审计结果**:

| 数据类型 | 处理方式 | 评价 |
|----------|----------|------|
| 用户手机号 | 明文存储 | ⚠️ 应加密存储 |
| 飞书 Secret | 环境变量 | ✅ |
| LLM API Key | 环境变量 | ✅ |
| 数据库密码 | 连接字符串 | ⚠️ 需确保不在日志中打印 |
| 审计日志内容 | 明文存储 | ⚠️ 可能包含敏感信息 |

**P2 - 手机号未加密**: `User.phone` 以明文存储在数据库，银行等保三级要求个人敏感信息加密存储。

## 五、并发与性能分析

### 5.1 并发能力评估

**架构并发模型**:
```
FastAPI (asyncio) → 异步处理 HTTP 请求
  ↓
BackgroundTasks → 后台异步任务
  ↓
LLM Gateway (httpx.AsyncClient) → 异步 HTTP 调用 LLM
  ↓
SQLAlchemy (asyncpg) → 异步数据库查询
```

**理论并发能力**:
- FastAPI + uvicorn: 单进程 ~1000 QPS (无计算密集型操作)
- LLM 调用是主要瓶颈: 单次 2-15 秒
- 并发 LLM 调用受限于:
  - httpx 连接池大小（默认 100）
  - LLM 提供商 API 限流

**预估并发能力**:
| 场景 | 预估并发数 | 瓶颈 |
|------|------------|------|
| 纯 API 路由 | 500-1000 QPS | CPU |
| LLM 意图识别 | 10-20 QPS | LLM API 延迟 |
| RAG 检索 | 50-100 QPS | 向量计算 |
| Skill 执行 | 5-15 QPS | LLM + 数据库 |

### 5.2 异步任务实现分析

**Celery 配置**（`app/tasks/celery_app.py`）:

**Broker**: RabbitMQ (`pyamqp://guest@localhost//`)
**Backend**: Redis (`redis://localhost:6379/1`)

**问题诊断**:

**P2 - Celery 任务未充分利用**: 当前消息处理使用 `BackgroundTasks` 而非 Celery。`BackgroundTasks` 在 FastAPI 进程内执行，进程重启时未完成的任务会丢失。Celery 更适合持久化的异步任务。

**建议**: 将 Skill 执行（特别是耗时的周报生成、WBS 生成等 LLM 密集型操作）迁移到 Celery 任务。

### 5.3 缓存策略评估

**已实现的缓存**:
| 缓存项 | 实现 | TTL |
|--------|------|-----|
| WeCom access_token | 内存缓存 (`_access_token`) | 7200s (提前300s刷新) |
| 消息幂等 | Redis (`get_message_idempotency_service`) | 待确认 |
| LLM 使用统计 | 内存字典 (`_usage_stats`) | 无过期 |

**问题诊断**:

**P1 - 缺少知识库缓存**: RAG 检索每次都调用 LLM 生成 embedding（`llm_gateway.get_embedding(query)`），每次增加 100-500ms 延迟和额外 Token 成本。相同查询应缓存 embedding 结果。

**P2 - LLM 使用统计内存泄漏风险**: `_usage_stats` 字典按用户 ID 追加记录，无上限、无清理机制，长期运行可能导致 OOM。

### 5.4 资源瓶颈识别

**已识别瓶颈**:

1. **LLM API 调用** (主要瓶颈)
   - 每次意图识别: ~2000 tokens, 1-3s 延迟
   - 每次 Skill 执行: ~4000-8000 tokens, 3-15s 延迟
   - 每次 RAG 检索: 1 次 embedding + LLM 生成

2. **RAG 向量计算**: Python 内存计算，未利用 pgvector 索引

3. **httpx 连接池**: 默认连接池可能不足（默认 100 连接）

4. **数据库连接池**: 未显式配置连接池大小，使用 SQLAlchemy 默认值（通常 5-10）
