# PM Digital Employee - 全流程自动化测试方案

## 一、测试概述

### 1.1 测试目标
- 确保PM Digital Employee系统核心功能稳定运行
- 验证飞书集成的完整性
- 检验Skill系统、编排引擎、数据管理的正确性
- 目标覆盖率：80%+

### 1.2 测试范围

| 测试层级 | 覆盖模块 | 优先级 |
|---------|---------|--------|
| 单元测试 | Skills、Orchestrator、Lark集成、Utils | P0 |
| 集成测试 | API端点、数据库操作、消息流程 | P1 |
| E2E测试 | 用户消息→Skill执行→结果返回 | P2 |

### 1.3 测试环境
- Python 3.11
- pytest + pytest-asyncio
- PostgreSQL 15 (mock)
- Redis (mock)
- 飞书API (mock)

## 二、测试策略

#### 2.1.3 Lark集成测试

- 测试消息发送API

#### 2.1.4 Utils测试
- datetime_utils时间处理
- crypto_utils加密解密

### 2.2 集成测试策略

#### 2.2.1 API端点测试
- Lark Webhook接收事件

- 数据管理API

#### 2.2.2 数据流程测试
- 消息接收→处理→响应完整流程

- 模拟飞书用户发送消息
- 完整流程验证
- 结果展示检查

## 三、测试用例设计

### 3.1 Skills单元测试用例

| Skill | 测试用例 | 验证点 |
|-------|---------|--------|
| ProjectOverviewSkill | execute无project_id | 返回错误 |
| ProjectOverviewSkill | execute有效project_id | 返回项目数据 |
| WeeklyReportSkill | execute无参数 | 返回错误或默认 |
| RiskAlertSkill | execute正常 | 返回风险列表 |
| CostMonitorSkill | execute正常 | 返回成本数据 |

### 3.2 Orchestrator测试用例

| 场景 | 输入 | 预期输出 |
|------|------|---------|
| 新意图识别 | "查看项目状态" | IntentResult.matched=True |
| 对话状态IDLE | 任意消息 | 触发意图识别 |
| 对话状态PARAM_COLLECTING | 参数输入 | 收集参数继续 |
| 对话状态CONFIRMATION_PENDING | "确认" | 执行Skill |

### 3.3 Lark集成测试用例

| 测试项 | 输入 | 预期结果 |
|-------|------|---------|
| 签名验证 | 有效签名 | 验证通过 |
| 签名验证 | 无效签名 | 验证失败 |
| Webhook challenge | challenge请求 | 返回challenge响应 |
| 消息接收 | im.message.receive_v1 | 处理成功 |

### 3.4 Utils测试用例

| 模块 | 测试项 | 验证 |
|------|-------|------|
| datetime_utils | format_datetime | 格式正确 |
| datetime_utils | get_week_range | 返回正确周范围 |
| crypto_utils | encrypt/decrypt | 加解密一致 |
| validators | is_valid_uuid | UUID格式验证 |
| validators | is_valid_email | 邮箱格式验证 |

## 五、测试执行计划

1. **Phase 1**: 运行现有测试，识别问题
2. **Phase 2**: 补充缺失测试用例
3. **Phase 3**: 执行全量测试
4. **Phase 4**: 修复发现的问题
5. **Phase 5**: 回归测试验证
6. **Phase 6**: 生成测试报告

## 六、Bug修复策略

- 发现问题立即定位根因
- 优先修复影响核心功能的Bug
- 修复后必须执行回归测试
- 记录Bug详情和修复方案

## 七、测试报告结构

1. 测试执行概况
2. 测试覆盖率统计
3. Bug发现与修复记录
4. 回归测试结果
5. 测试结论与建议
