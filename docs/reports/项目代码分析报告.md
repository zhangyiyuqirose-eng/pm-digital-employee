### 2.6 编排层 (`app/orchestrator/`)

**核心流程** (`orchestrator.py`):
```
1. 接收飞书消息 → process_lark_message()
2. 构建用户上下文 → _build_user_context()
3. 获取/创建对话会话 → DialogStateMachine
4. 根据状态路由:
   - IDLE → _handle_new_intent() → 意图识别 → Skill执行
   - PARAM_COLLECTING → _handle_param_collection()
   - CONFIRMATION_PENDING → _handle_confirmation()
5. 发送响应 → LarkService
```

**对话状态机**:
- `IDLE`: 等待新意图
- `PARAM_COLLECTING`: 参数收集
- `CONFIRMATION_PENDING`: 等待确认
- `EXECUTING`: 执行中
- `COMPLETED`: 完成

## 四、核心业务逻辑与数据流

### 4.1 消息处理主流程

```mermaid
sequenceDiagram
    participant Lark as 飞书
    participant Webhook as lark_webhook.py
    participant Orchestrator as orchestrator.py
    participant Context as ContextService
    participant Intent as IntentRouter
    participant Skill as SkillRegistry
    participant DB as PostgreSQL
    participant Redis as Redis

    Lark->>Webhook: POST /api/v1/lark/event
    Webhook->>Webhook: 验签 + 去重
    Webhook->>Orchestrator: process_lark_message()
    Orchestrator->>Context: build_user_context()
    Context->>DB: 查询用户/项目绑定
    Context->>Redis: 缓存上下文
    Orchestrator->>Intent: recognize_with_context()
    Intent->>Intent: LLM意图识别
    Orchestrator->>Skill: execute_skill()
    Skill->>DB: 数据操作
    Skill->>Lark: 发送响应卡片
```

### 4.2 项目隔离安全流程

```mermaid
flowchart TD
    A[飞书消息] --> B{群聊?}
    B -->|是| C[查询GroupProjectBinding]
    B -->|否| D[查询用户可访问项目]
    C --> E{绑定存在?}
    E -->|否| F[抛出GroupNotBoundError]
    E -->|是| G[获取project_id]
    D --> H[选择project_id]
    G --> I[查询UserProjectRole]
    H --> I
    I --> J{角色存在?}
    J -->|否| K[抛出ProjectAccessDeniedError]
    J -->|是| L[构建UserContext]
    L --> M[所有数据操作强制project_id过滤]
```

### 4.3 Skill执行流程

```
1. IntentRouter识别意图 → 返回IntentResult（skill_name, params）
2. SkillRegistry获取Skill类
3. 创建Skill实例（传入Manifest、Context、Session）
4. 调用execute()
5. 返回SkillExecutionResult
6. ResultFormatter格式化
7. LarkService发送响应
```
